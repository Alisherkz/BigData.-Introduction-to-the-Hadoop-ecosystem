Задача будем играться с форматами и самим файлом где хранятся данные.

1. Создаем таблицу. Код нж
> insert into dbname.tablename
< 1 as id,
< "myText" as str;

	Факт. Если проверить папку где хранятся данные, можно заметить что создался новый файл 0000 типо такого
с данными. Если открыть файл то там должный находится данные который мы записовали.

2. Проверяем саму таблицу создался ли?.
> select * from tablename;

3. Добавляем инфо к таблицу(Базу).
> insert into dbname.tablename
< 2 as id,
< "test1" as str;	# Можно несколько раз добавить.

	Факт. Каждый раз когда вы добавляете инфо к таблицу, Файл обнавляется и создается 
копии файла, 0000_copy1, копи2 и т.д.

4. Меняем называние колонки str на str_new
> alter table dbname.tablename change str str_new string;

	Факт. Когда меняем называние колонки, все данные который были в этом колонке они исчезает, то есть это Null
Если поставить обратно старое называние колонки, то данные вернется.



 - - - - - - - - - - - - - - Нужно узнать. Когда мы записываем новые данные то создается новый файл с этими данными,
Вопрос такой, то есть каждый раз когда мы записываем данные создается ли файл каждому строку или когда каждый раз когда мы записываем
данные


5. Мы можем сжать файл, код нж
> Set parquet.compression=Snappy;
> Set parquet.compression=GZIP

	Факт. Для проверки нужно написать
> parquet-tool meta hdfs://novalocal:8020/../../dbname/tablename/000000_0_copy_num типо так нужно написать
Самом внизу должен быть сообщение об архиве.
!---> id:	INT32 GZIP DO:0 FPO:4 SZ:61/43/......
!---> str:	BINARY GZIP DO:0 FPO:65 SZ:62/44/.....

	Нужно знать. Мы комперссируем файл, есть несколько типов компресси такие как GZIP, BZIP2, SNAPPY, LZO и т.д.
---> Gzip: утилита сжатия, принятая в проекте GNU, Gzip (сокращение от GNU zip) генерирует сжатые файлы с расширением .gz. Вы можете 
использовать команду gunzip для распаковки файлов, которые были созданы с помощью ряда утилит сжатия, включая Gzip.

---> Snappy: кодек Snappy от Google обеспечивает умеренную степень сжатия, но высокую скорость сжатия и распаковки. 
(Фактически, он имеет самую высокую скорость распаковки, что делает его очень желательным для наборов данных, которые, вероятно,
будут часто запрашиваться.)
